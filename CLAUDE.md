# FinQuery - Project Context

## What This Is
FinQuery is a **grounded personal-finance Q&A agent**: users ask questions about their own bank
transactions ("How much did I spend on food in June?") and core personal-finance concepts ("What is
the 50/30/20 rule?") and get exact, SQL-computed, provenance-backed answers, with a hard guarantee
that **raw PII never enters LLM context** (reversible pseudonymization layer, 0% leakage target).

It is being built as the **capstone project for the AIEngg.dev "AI Engineering Cohort: RAG & Agents"**.
After the capstone it will be integrated into CraftMyMoney (the author's personal-finance platform,
lives at `~/CraftMyMoney/code/cmm`) as its RAG-based finance bot feature. Building fresh for the
capstone and integrating into CMM afterwards is legitimate; the cohort's "reused projects = zero"
rule forbids submitting *pre-existing* work, not future reuse.

## The Frozen Spec
`docs/capstone_design_doc.md` is the **approved-for-build design doc**. Read it before doing
anything. It defines: problem statement, both architectures, PII layer, retrieval design, eval
criteria, framework justification, timeline. Do not re-litigate decisions recorded there.
`docs/capstone_design_doc_krantikumar.pdf` is the exact version submitted to the cohort.

Architecture in one paragraph: two approaches built and compared (mandatory comparative eval).
**Approach A** = vanilla RAG baseline, everything (KB + serialized transactions) embedded in
pgvector, hypothesis: fails on aggregations. **Approach B** = single tool-calling agent (ReAct loop,
Pydantic AI, gpt-4o-mini) with 6 typed tools: 5 SQL tools (`spending_by_category`,
`budget_vs_actual`, `top_merchants`, `income_summary`, `search_transactions`) + `search_finance_kb`
(hybrid dense + BM25 merged with RRF over ~500 KB chunks in pgvector). Numbers are computed by SQL,
never generated by the LLM. Single Postgres instance for rows, vectors, and sparse index.

## Decisions Made in Design Discussions (not all visible in the doc)
- **Composite questions** ("After paying my EMIs I can't save my salary sufficiently") are an
  **emergent capability of the ReAct loop**, never a listed feature. They prove vanilla RAG's
  structural limits and justify the agent. 5 composite questions exist in the golden set.
- **Refusal guard is a specificity filter, not a topic filter.** General principles applied to the
  user's numbers = education (allowed). Product-level recommendations ("should I buy X fund?") =
  refused (SEBI RIA regulation in India).
- **PII layer**: forward pseudonymization (real -> fake via per-user mapping table) is core and
  applies to BOTH approaches. Detection = regex (account numbers, phones, UPI VPAs) + known-values
  list. Free-text **name NER is documented out of scope**. **Reverse mapping is an optional
  extension**, required only if counterparty-name questions are added. Prior art: Microsoft Presidio
  (acknowledge, don't import).
- **Rejected with reasons** (these carry Decision Reasoning rubric points, keep them documented):
  Graph-RAG/KAG, multi-agent, map-reduce summarization, memory/multi-turn, re-ranking and HyDE
  (deferred until evals expose misses they would fix), LangGraph (hides the ReAct loop being
  evaluated), smolagents (code-agent = wrong risk profile for finance).
- **Rejected: classifier-router in front of RAG/agent.** Tool selection IS the routing; composite
  questions prove a hard router is harmful.
- **Rejected: self-hosted open-source models.** GPU cost, and the cohort provides the OpenAI key.
- **Fixed category taxonomy** (Food & Dining, Transport, EMI, Rent, Utilities, Salary, ...) assigned
  at seed time and included in the agent system prompt. Transaction categorization itself is out of
  scope.
- **Chunking**: ~400-token chunks, 15% overlap. **Index**: exact/flat; HNSW/IVF rejected as scale
  solutions a ~500-chunk corpus does not need.
- **Eval matrix**: {Vanilla RAG, Agent + dense, Agent + hybrid-RRF} x 58 golden questions in 5
  buckets (aggregation 15, lookup 10, education 15, refusal 13 incl. 3 prompt-injection probes,
  composite 5). Ground truth precomputed via SQL against the seeded DB. Thresholds: >=95% numeric exact-match on aggregations,
  >=90% refusal on probes, 0% PII leakage (deterministic scan of every LLM-bound payload against the
  real-PII mapping table, no judge). LLM-as-judge = gpt-4o (judge/actor separation from gpt-4o-mini).

## Hard Boundaries
- **Never connect to CraftMyMoney's database or import CMM code/data.** This repo is standalone:
  own `docker-compose.yml`, synthetic seed (1 user, ~6 months of realistic Indian transactions),
  reproducible with one `docker compose up`. Mentors must be able to run it cold.
- **Keep CMM business IP out**: FinHealth pillar weights, scoring tables, and framework internals
  stay in the CMM repo, never here.
- **Copyright**: do NOT ingest copyrighted books (e.g., Monika Halan's "Let's Talk Money"). KB =
  public RBI investor-awareness booklets, SEBI/NCFE investor-education material, plus short original
  articles written for this project.
- **Synthetic data only.** Seed narrations must contain realistic fake PII (VPAs, phone numbers,
  account refs) so the pseudonymization layer has something real to catch and the 0%-leakage metric
  is meaningful.

## Build Order (aligned to doc timeline, resequenced around the API key)
Key-independent work first:
1. Repo scaffold: docker-compose (Postgres + pgvector), FastAPI skeleton, `/ask` stub.
2. Synthetic seed script + fixed taxonomy. Ensure interesting ground truth exists (e.g., at least
   one over-budget category).
3. Golden question set (~55) + SQL-precomputed ground truth. Highest rubric value; do this early,
   it exposes seed-data gaps.
4. PII layer + unit tests (pure regex + mapping table, no LLM needed).
5. KB collection + chunking code (embedding waits on the key).

Key-dependent: embed KB -> vanilla RAG baseline -> agent + 6 tools -> hybrid ablation -> eval
harness -> minimal chat UI (suggested-question chips + free text, single-turn) -> demo video + docs.
