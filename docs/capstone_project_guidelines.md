Capstone Project Specification
AI Engineering Cohort: RAG & Agents
Objective
This capstone requires participants to validate their problem definition, architecture, and evaluation methodology before implementation. Participants must avoid common pitfalls like vague objectives or lack of objective performance metrics.
Design Doc Requirement
Approval of the design doc is a prerequisite for receiving an OpenAI API key.
Required Design Doc Sections
Problem Statement: A specific, narrow definition of the challenge being addressed.
Architecture: A high-level overview of the system design (RAG, ReAct, etc.).
Evaluation Criteria: Specific quantitative and qualitative success metrics.
Framework Justification: Selection of frameworks (e.g., Pydantic AI, smolagents, LangGraph) with reasoning.

Submission Channel: Submission Form 
Group Allocations: Group Allocations
Project Requirements
Functional systems must include:

Data Surface: A knowledge base or tool-use interface that the system actively queries.
Core Flow: An implementation of a retrieval or orchestration flow, such as a RAG pipeline, a ReAct-style loop, or Multi-Agents System.
Comparative Evaluation: A mandatory evaluation comparing at least two distinct approaches (e.g., Vanilla RAG vs. Graph RAG vs. Agent), with a justified final choice.
Failure Analysis: A README section documenting attempted approaches, failures, and subsequent pivots.
User Interface (Optional): A functional interface that allows external users to interact with the system.

Common Failure Modes to Avoid
Vague Problem Statements: "An AI that helps with everything" is not a project.
Architecture-First Bias: Selecting complex frameworks before establishing a baseline or understanding data.
Complexity Bias:  Example, Skipping a simple baseline before adding advanced retrieval techniques.
